# Multi-Cloud Unified Dashboard — Hackathon MVP

A read-only dashboard that unifies resource inventory and cost data from **AWS**, **GCP**, and **Azure** into a single view.

Backend: **FastAPI** · Frontend: **Next.js + Tailwind CSS**

See [`SRS.md`](./SRS.md) for full requirements and [`PRD.md`](./PRD.md) for product scope, user stories, and timeline.

## 2. Prerequisites

- **Python** 3.11+
- **Node.js** 18+ and npm
- Cloud CLI credentials already configured (at least one cloud recommended; all three ideal):
  - AWS: `aws configure`
  - GCP: `gcloud auth application-default login`
  - Azure: `az login`
- AWS Cost Explorer, GCP Cloud Billing, and Azure Cost Management enabled on the respective test accounts **before** you start (Cost Explorer especially — it can take ~24h to populate after first enabling).

## 3. Backend Setup (FastAPI)

```bash
cd backend
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

Copy the example env file and fill in credentials for whichever clouds you're using live:

```bash
cp .env.example .env
```

`.env` fields:
```
AWS_ACCESS_KEY_ID=
AWS_SECRET_ACCESS_KEY=
AWS_REGION=us-east-1

GCP_SERVICE_ACCOUNT_JSON_PATH=./gcp-service-account.json
GCP_BILLING_ACCOUNT_ID=

AZURE_CLIENT_ID=
AZURE_TENANT_ID=
AZURE_CLIENT_SECRET=
AZURE_SUBSCRIPTION_ID=
```

If any cloud's credentials are missing or a live call fails, that adapter automatically falls back to its mock JSON file under `backend/mocks/` and is flagged `"source": "mock"` in the API response — the app will still run end-to-end.

Run the API:
```bash
uvicorn main:app --reload --port 8000
```

API docs (auto-generated by FastAPI) available at:
```
http://localhost:8000/docs
```

## 5. Key API Endpoints

| Endpoint | Description |
|---|---|
| `GET /api/health` | Per-provider adapter status: `live`, `mock`, or `error` |
| `GET /api/resources` | Unified resource list. Query params: `provider`, `type`, `region` |
| `GET /api/costs` | Unified cost breakdown. Query params: `provider`, `granularity` |
| `GET /api/costs/summary` | Combined total spend + per-provider subtotal |

Example:
```bash
curl "http://localhost:8000/api/resources?provider=aws&type=compute"
curl "http://localhost:8000/api/costs/summary"
```

## 6. How to Navigate This Repo (Quick Orientation)

1. Start with **`PRD.md`** to understand what's in scope vs. explicitly out of scope for this MVP.
2. Read **`SRS.md`** §6 for the exact unified schema every adapter must conform to — this is the contract the whole system depends on.
3. In `backend/adapters/`, each file is self-contained: it calls one cloud's SDK, maps the response to the unified schema, and falls back to `backend/mocks/*.json` on failure. Start here if you're adding a new resource type or fixing a broken cloud call.
4. `backend/services/normalizer.py` is where all three adapters' outputs get merged — this is the only place that should combine cross-cloud data.
5. `backend/routers/` exposes the normalized data as REST endpoints — this is the only layer the frontend talks to.
6. In `frontend/components/`, each component maps directly to one part of the UI described in PRD §7 (SummaryCard = combined total, CostBreakdownChart = per-cloud chart, ResourceTable = filterable list, ProviderStatusBadge = live/mock indicator).

## 7. Demo Script (Suggested)

1. Open dashboard — point out the combined monthly spend number first (the "money shot").
2. Point out the per-cloud breakdown chart underneath.
3. Filter the resource table by provider, then by type, to show unification in action.
4. Point out the live/mock status badges — be upfront about which clouds are live vs. seeded for the demo.
5. Open `/docs` briefly to show the underlying unified API contract.

## 8. Known Limitations (By Design, See PRD §5.2 / §8)

- Read-only — no provisioning or write actions against any cloud.
- No auth/multi-user support.
- No persistent historical data store (in-memory/SQLite only).
- Cost figures reflect each cloud's native billing latency and are aggregated at the service/provider level, not guaranteed per-resource.
